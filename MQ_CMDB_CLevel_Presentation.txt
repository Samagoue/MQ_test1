SLIDE 5 — HOW IT WORKS
────────────────────────────────────────────────────────────────────────────────

Three inputs. One output.

  INPUT:
    ├── CMDB asset export      (all MQ assets)
    ├── Country code reference (37 countries)
    └── Pattern rules          (how queues map across countries)

  OUTPUT:
    └── Confluence documentation — published automatically

  The pipeline is intelligent:
  • It recognises naming conventions
  • It infers relationships between assets
  • It groups assets by application, country, and function
  • No manual mapping required — the architecture is derived from the data


────────────────────────────────────────────────────────────────────────────────
SLIDE 6 — THE CROSS-COUNTRY ASSOCIATION (UNIQUE CAPABILITY)
────────────────────────────────────────────────────────────────────────────────

This is something no tool was doing before.

The system automatically identifies how a queue in the UK maps to its
counterpart in Argentina, France, or Kuwait — by pattern recognition:

  • ORG_AP_COUNTRYUK  ↔  ORG_AP_COUNTRYFR     (two-character code variant)
  • ORG.ARGENTINA     ↔  ORG.KUWAIT            (full country name variant)

Why this matters:

  • Cross-country messaging failures are the hardest to diagnose
  • Without this map, no engineer knows what they should be looking at
  • With it, root cause identification drops from hours to minutes
  • 37 countries covered. Automatically. Every time.
